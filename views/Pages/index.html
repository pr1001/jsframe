jsFrame - what's that?

jsFrame is a web application framework written entirely in Javascript and HTML by Peter Robinett over the course of a lazy Sunday. The framework was simply intended as a programming exercise and is only at version #{vars.version}, so it's surely not production-ready! That being said, I'm putting it out there in the hope that other people can benefit from it and if it strikes there fancy can grow beyond its modest origins.

Yes, I know there's another jsFrame. But I had chosen the name when I found them and they don't have any code to download, so there!

Download

You can download the latest source code at http://github.com/pr1001/jsframe/tree/master or clone it using git via git://github.com/pr1001/jsframe.git.

Details

jsFrame uses the MVC pattern, which means that it should be easy to get going and write nice code. It makes heavy use of Prototype, so if you're used to it you'll feel right at home. jQuery, MooTools, and YUI fans are welcome to take a shot at forking the code or abstracting out Prototype! I've spent the last year working in CakePHP, so I'm sure its styles (and by extension, Ruby on Rails) have rubbed off on jsFrame's design. Like those framework, jsFrame favors convention over configuration.

jsFrame tries to be efficient as possible and load files only when needed, while still giving you the benefits of playing within the jsFrame namespace. All hyperlinks are automatically sent through jsFrame's routing system without the need for a page reload. The routing system is moderately flexible and there is an Apache .htaccess configuration enabling pretty urls included.

To-Do

Where to start! There are lots of things that need serious work:

  • Add datastores to models - Currrently there are no database backends and both local (HTML 5 sqlite-based querying) and remote (AJAX-based querying) would be useful.
  • Improved routing - While both path (e.g. "/") and parameter based (e.g. "controller=Pages&action=index") routing work, the regexes and the parameter extraction process need work.
  • Error handling - The code currently falls apart with anything but the most minor error. Not good!
  • Logging - Ways to log errors and user actions would be very useful. Logging to the browser (e.g. Firebug's console.log() method) or to disk (e.g. AJAX-based POSTs) would be nice.
  • Get rid of all those eval()'s - I like variable variable names and couldn't figure out an easy way to use them other than by using lots of eval() statements. I'm sure a javascript guru can correct my errant ways.
